🤖 THE QPAY AGENTIC MASTER PLAN
Target: Transform qpay.qubickle.com from a buggy script into a highly scalable, "Stripe-like" enterprise payment orchestrator.

🌟 The Core Vision (Context for the AI Agent)
The Mission: QPay is bridging the gap for merchants in South Asia (e.g., Bangladesh) by providing a unified checkout experience for MFS (bKash, Nagad) and Banks.

The Creative Challenge: QPay is currently reliant on Android SMS-scraping. The AI agent must help architect an Adapter Pattern. We need to seamlessly transition from SMS-verification to Direct Bank API verification without the merchant ever noticing the switch.

The Architecture Goal: We are decoupling the monolithic structure. Eventually, api.qpay.qubickle.com (headless, ultra-fast transaction processing) will be completely separate from dashboard.qpay.qubickle.com (heavy analytics and UI).

🛡️ The "Budget-Safe, Zero-Downtime" Rules of Engagement
Dear AI Agent, you must strictly follow these rules:

Atomic Micro-Tasks: You will only perform one complete, testable task at a time. Never start a massive refactor that requires 50 back-and-forth prompts. If the daily token budget runs out, the current deployed code must compile and run flawlessly.

Append, Don't Replace: When adding new architecture (like the new API routes), do not delete the old legacy routes yet. Create api/v1/ routes alongside the legacy system.

Shadow Deployment: When building the new "Double-Entry Ledger," build it to run alongside the old balance update system. We will write to both databases simultaneously until the new ledger is proven stable.

No "Null" States: Never output placeholders like // Add logic here later. Write complete, working functions for the specific micro-task requested.
